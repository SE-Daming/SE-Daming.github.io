优化器如何评估走全表还是走索引

2025年03月03日 留言

优化器作用

根据是否使用临时表、是否排序、扫描行数去选择使用哪个索引、多个join时各个表的连接顺序

优化器逻辑

优化器要统计索引基数、预估语句扫描行数

1、基数cardinality：一个索引上不同值得个数，基数越大索引区分度越好、优化器更偏向使用

如何统计基数：采样统计。默认选择N个数据页计算平均值*索引页面数；当变更的数据超过1/M时重新统计

【了解】

在 MySQL 中，有两种存储索引统计的方式，可以通过设置参数 innodb_stats_persistent 的值来选择：

  • 设置为 on 的时候，表示统计信息会持久化存储。这时，默认的 N 是 20，M 是 10。
  • 设置为 off 的时候，表示统计信息只存储在内存中。这时，默认的 N 是 8，M 是 16。

1、force index强制纠正索引

2、修改SQL语句，引导优化器使用期望的索引

MySQL为什么不用二叉树、跳表，而是B+树

2025年03月02日 留言

B+树

1、B+树是多路平衡搜索树，每个节点存储多个键值和指针，非叶子节点只存放索引键和指针，叶子节点存放数据且节点通过双向链表连接【范围查询高效】

2、3层B+树可支持千万级数据，查询仅需3次磁盘io【高效磁盘io】

3、更稳定的查询效率

红黑树

1、红黑树高，io次数多

2、红黑树范围查询效率低下

3、节点存储数据+指针，内存占用高

二叉树

1、退化为链表风险

2、节点利用率低，只能存储一个键值和两个指针，无法充分利用磁盘块空间

3、需要通过中序遍历进行范围查询，效率低下

跳表

1、千万级别数据量高度可达20多层，每次查询触发多次随机磁盘io

2、节点分布在不同数据页，磁盘寻址开销大

适用场景

  • B+树：读多写少、范围查询频繁的数据库场景（如MySQL、Oracle）
  • 跳表：内存数据库（如Redis ZSET）、写密集型存储引擎（如RocksDB）
  • 二叉树：内存中小规模数据、无需范围查询的场景

http3协议

2025年02月28日 留言

http2的缺点

HOL队首阻塞

http2中多个请求跑在一个TCP连接中，如stream2丢失，即使收到了stream3、4也会被阻塞直到stream2重传

http2只在应用层解决了HOL、传输层仍然存在

TLS握手时延迟

需要3个RTT时延

网络迁移需要重新连接

从4G变成WiFi需要重新建立TCP与TLS握手

QUIC协议特点

http3不仅仅是将传输协议换成了UDP，还基于UDP在应用层实现了QUIC协议，它拥有类似TCP的连接管理，拥塞窗口，流量控制的特性，相当于将不可靠的UDP变成可靠的了

无HOL

QUIC上的多个stream间没有影响，一个stream中的数据丢失了不影响其他stream

更快的连接建立

QUIC协议只需要一次RTT，QUIC内包含了TLS

连接迁移

QUIC协议不像TCP使用四元组（源IP，port，目标ip，port）绑定连接，而是用连接id来标记通信的两个端点，当IP变化后只要连接id和TLS密钥等不变就可以无缝复用原连接，消除重连成本，起到连接迁移的功能

HTTP3协议

HTTP3在HTTP的变化

HTTP2的二进制帧需要定义stream，HTTP3直接用QUIC里面的stream，二进制帧的结构变简单了；

HTTP3的头部压缩算法换成了QPACK，相较于HTTP2的HPACK

介绍G1垃圾回收器

2025年02月26日 留言

设计背景

替代CMS，解决其内存碎片、不可控停顿时间、并发模式失败

改进目标：适应大内存堆、可控的停顿时间、无内存碎片

核心特点

分区模型

将堆分为多个大小相同的Region（1MB、32MB），每个Region都可以是Eden、Survivor、Old或Humongous

分代收集

逻辑上区分年轻代和老年代，支持混合回收，同时回收年轻代和老年代

可预测停顿模型

通过参数MaxGCPauseMillis设置最大停顿时间，优先回收垃圾比例高的Region

并行与并发

并行：多线程执行GC（如标记、复制），减少STW时间

并发：并发标记阶段和应用线程交替执行，减少停顿

工作流程

年轻代回收

Eden区满时触发：暂停所有应用线程（STW）、将Eden区存活对象复制到Survivor区、年龄递增、清空Eden

混合回收

  1. 初始标记（Initial Marking）：
    • 标记GC Roots直接关联的对象，需STW，但时间极短（与Young GC同步触发）
  2. 并发标记（Concurrent Marking）：
    • 遍历堆，标记存活对象，与用户线程并发执行。
    • 使用SATB（Snapshot-At-The-Beginning）算法，记录标记开始时的对象快照，处理并发修改
  3. 最终标记（Final Marking）：
    • 处理SATB队列中的引用变更，修正标记结果，需STW
  4. 筛选回收（Evacuation）：
    • 根据Region的回收价值（垃圾比例）和成本（复制时间），选择Region构成回收集。
    • 将存活对象复制到新Region，清空旧Region，需STW

关键数据结构

记忆集、卡表

优缺点

优点

可控停顿：适合对响应时间敏感（实时系统）

高吞吐量：并行回收充分利用多核CPU

无内存碎片：用标记整理算法

缺点

内存开销：维护记忆集和卡表额外占内存

小内存性能差：CMS可能更高效

Full GC风险：当大对象分配失败或者并发回收速度不足时，退化成单线程Full GC

适用场景

大内存堆、低延迟需求、长期运行应用
