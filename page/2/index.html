零拷贝

2025年03月06日 留言

为了减少数据拷贝和优化系统调用的一种技术，可以将文件传输性能提高一倍以上

这个概念针对io工作方式来说的

传统的io方式：涉及到用户态和内核态，

【数据读取】在内核态中，DMA读取磁盘文件到内核缓冲区，然后CPU将内核缓冲区的数据读取到用户态的用户缓冲区。

【数据写入】CPU读取用户缓冲区文件到内核缓冲区，DMA读取到内核态磁盘文件

四次上下文切换（用户态《=》内核态）、四次数据拷贝

零拷贝技术：只涉及内核态

通过一次系统调用sendfile合并了磁盘读取和网络发送两个操作、DMA直接完成内存和设备的数据传输，无需CPU参与

两次DMA拷贝、一次内核态拷贝。

select,poll,epoll的区别

2025年03月06日 留言

三者区别

select用户态维护一个FD集合保存建立连接的socket，然后将集合复制到内核态，内核态的遍历集合标记就绪状态，然后拷贝回用户态，用户态遍历集合找到就绪的socket；
poll和socket流程一样，唯一不同的是select集合是bitsmap（最多1024个）、poll是一个链表形式组织的动态数组（支持更多连接 ）；
epoll内核态维护红黑树保存socket，用户态建立socket连接时只需要传入一个socket给内核态，减少了拷贝和内存分配。另外维护一个链表记录就绪的socket，当用户态查看就绪socket，内核态返回其就绪socket。

epoll：epoll_create epoll_ctl epoll_waitl

边缘触发和水平触发

水平触发：只要有socket就绪，就持续通知应用程序直至处理

边缘触发：当socket就绪时只通知一次

synchronized

2025年03月05日 留言

sync是Java中实现线程同步的关键字，底层基于Monitor机制，通过JVM内部的数据结构和锁优化策略实现线程互斥，保证线程安全【黑马JUC 】

Java中每个对象都有一个对象头，包含Mark Word和Klass Word、（array length），每一个对象都有一个monitor与之对应

底层原理

原子性

可见性

获取锁时强制从主内存读写；释放锁时刷新到主内存；内存屏障

有序性

锁的互斥性等效单线程执行，保证有序性

可重入

偏向锁不记录重入次数；

轻量级锁通过Lock Record的个数记录；

重量级锁通过monitor对象的recursion字段记录

不可中断

锁升级

无锁 —>偏向锁 —>轻量级锁 —>重量级锁

偏向锁（JDK15后被废弃）

首次有线程访问，并且无竞争线程。

加锁时Mark Word记录线程id并把锁状态标志设为01；

当有其他线程竞争时升级为轻量级锁

轻量级锁

一个对象有多个线程访问，但是访问时间是错开的，不存在竞争，用轻量级锁优化；

加锁：线程在栈帧中创2建Lock Record锁记录，通过CAS尝试把锁记录地址替换为Mark Word，并把对象头的Mark Word替换成锁记录；成功：则获取到轻量级锁，锁状态为00。失败：1、如果是自己持有锁，是自己则再创建一个锁记录，锁记录地址为null；2、不是自己则有其他线程持有轻量级锁，进入锁膨胀过程（先自旋、失败次数一定时升级为重量级锁）；

解锁：当锁记录地址为null则清理掉，锁记录减一；不为null则CAS将Mark Word恢复给对象头，1、成功则解锁 2、失败则锁升级为了重量级锁，进入重量级锁解锁流程

重量级锁（锁膨胀）

加锁：当线程2尝试获取锁时获取失败，为对象申请monitor对象，并把对象头的锁记录替换成monitor的地址，后两位设为10。owner设为线程1，自己进入monitor的等待队列等待。并把

解锁：线程1解锁时发现解锁失败，按照monitor地址找到monitor，将owner置为null，唤醒等待队列的线程

锁优化

锁升级、锁消除、锁粗话、自适应自旋锁（锁升级的时候动态调整自旋次数）

锁代码块和方法的区别

字节码层面：代码块显式插入monitor-enter /exit指令；方法是隐式标记acc-synchronized指令

其他

monitor创建使用的时机

  • 当线程首次通过synchronized竞争对象锁时，若锁状态升级为重量级锁，JVM会为该对象关联一个Monitor对象。
  • 调用wait()、notify()等方法时，需要操作Monitor的等待队列（_WaitSet）或竞争队列（_EntryList），此时必须关联Monitor

反射

2025年03月03日 留言

反射性能开销高的原因

运行时类型检查

普通代码在编译期确定变量类型，反射在运行时进行类型检查，并且需要动态解析类型信息，增加了额外开销

间接方法调用和字段访问

调用方法时要 查找方法描述符、检查参数类型并通过动态调度执行方法；访问字段时需要查找字段偏移量进行读取或写入，比直接访问字段更慢，因为直接访问通过内存布局直接计算字段地址

额外内存分配和垃圾回收压力

反射会创建临时对象。频繁反射会导致大量内存分配和释放，增加内存管理负担。可能导致STW

反射是什么

运行时动态获取类的信息，包括构造函数、字段、方法，允许开发者在不知道类结构的情况下动态创建对象、调用方法和访问字段

应用场景

bean依赖注入；ORM映射；动态代理；单元测试；插件化开发；注解；数据库驱动

根据是否使用临时表、是否排序、扫描行数去选择使用哪个索引、多个join时各个表的连接顺序

优化器逻辑

优化器要统计索引基数、预估语句扫描行数

1、基数cardinality：一个索引上不同值得个数，基数越大索引区分度越好、优化器更偏向使用

如何统计基数：采样统计。默认选择N个数据页计算平均值*索引页面数；当变更的数据超过1/M时重新统计

【了解】

在 MySQL 中，有两种存储索引统计的方式，可以通过设置参数 innodb_stats_persistent 的值来选择：

  • 设置为 on 的时候，表示统计信息会持久化存储。这时，默认的 N 是 20，M 是 10。
  • 设置为 off 的时候，表示统计信息只存储在内存中。这时，默认的 N 是 8，M 是 16。

MySQL为什么不用二叉树、跳表，而是B+树

2025年03月02日 留言

B+树

1、B+树是多路平衡搜索树，每个节点存储多个键值和指针，非叶子节点只存放索引键和指针，叶子节点存放数据且节点通过双向链表连接【范围查询高效】

2、3层B+树可支持千万级数据，查询仅需3次磁盘io【高效磁盘io】

3、更稳定的查询效率

红黑树

1、红黑树高，io次数多

2、红黑树范围查询效率低下

3、节点存储数据+指针，内存占用高

二叉树

1、退化为链表风险

2、节点利用率低，只能存储一个键值和两个指针，无法充分利用磁盘块空间

3、需要通过中序遍历进行范围查询，效率低下

跳表

1、千万级别数据量高度可达20多层，每次查询触发多次随机磁盘io

2、节点分布在不同数据页，磁盘寻址开销大

适用场景

  • B+树：读多写少、范围查询频繁的数据库场景（如MySQL、Oracle）
  • 跳表：内存数据库（如Redis ZSET）、写密集型存储引擎（如RocksDB）
  • 二叉树：内存中小规模数据、无需范围查询的场景
