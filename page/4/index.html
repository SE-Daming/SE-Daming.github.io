缓存双写一致性

2024年10月24日 留言

g

共三种方案

Cache Aside Read/Write Through Write Behind Caching

Cache Aside

数据库为主存储、缓存为提升效率

由缓存的调用者在更新数据库的时候更新缓存

工作原理

  • 读取操作：首先尝试从缓存中获取数据，如果缓存命中，则直接返回；否则，从数据库中读取数据并将其放入缓存，最后返回给客户端。
  • 更新操作：先更新数据库再删除缓存

为什么先更新数据库

如果先删除缓存后更新数据库、线程a删了缓存、线程b发现没有缓存则去读取数据库旧值重构缓存、线程a更新数据库=》不一致；

如果先更新数据库再删除缓存、如果没有缓存、线程a去读数据库旧值、线程b更新了数据库、删除了缓存、线程a重构旧缓存；

缓存操作的速度很快、不太可能出现第二种情况；而第一种就比较容易出现，MySQL速度没有这么快

为什么删除缓存

删缓存更加轻量；

更新的缓存可能长时间不被使用，从资源和性能角度考虑使用缓存的时候再构建，

删除缓存失败怎么办

消息队列的重试机制，将需要删除的放到消息队列，但是需要改造业务代码，对代码有侵入性

基于Canel订阅binlog日志发送到mq、消费者删除缓存，无侵入性但是增加了维护成本

主从同步延迟+读写分离

这种情况下采用更新数据库+删除缓存也会出现数据不一致；

可以采用延迟双删+订阅binlog+mq的方案

延迟双删

先删缓存、再更新数据库、再删缓存

延迟时间要大于线程 B 读取数据库 + 写入缓存的时间；

延迟时间要大于「主从复制」的延迟时间

Read/Write Through

只更新缓存、缓存同步去更新数据库

Write Behind Caching

只更新缓存、缓存异步去更新数据库

总结

采用微服务架构，模块化设计，各个服务之间通过RPC进行高效通信。我们使用MySQL的主从架构实现读写分离，提升性能，Redis作为缓存，MQ处理异步消息，Elasticsearch提供搜索功能。

在服务治理方面，我们使用Nacos作为配置和注册中心，Dubbo作为远程调用框架，Sentinel负责服务保护，Seata确保分布式事务的完整性，Gateway作为统一的API网关，处理外部请求。

订单模块

该方案的核心思想是，先创建死信交换机、队列和消费者，来监听死信消息。

然后创建定时过期的消息，比如订单支付的时间为 30min，则将消息的 TTL（最大存活时间）设置为 30min，将消息放到一个没有消费者消费的队列中，当消息过期后就会成为死信。

死信消息被重新发送到死信交换机，然后我们在死信队列中消费该消息，根据商品 ID 判断该商品是否被支付。
