介绍一下DRS

DRS系统是基于车云远程的诊断解决方案，具备整车级或单ECU级的如参数读写、故障读清、例程标定、动作测试和程序升级等传统常见诊断功能，并且支持基于云任务的车端主动诊断如车辆故障主动预警、关键数据采集、智能诊断等新型诊断方式，并在国内、欧洲等海外市场上量产使用

国际化方案

业务背景

静态资源–按钮、菜单：前端实现

动态资源—提示信息、诊断报告：后端实现

实现步骤

两种数据存储方案—yml资源文件、数据库

资源文件：易于管理维护、可和前端共享；但是难以应对大内容量和频繁更新

数据库：支持多语言内容的动态更新、不需要重新部署、适用频繁更新；存储和查询开销大、性能问题

我们的方案是用spring的i18n框架messageSource，（实现简单、快速上限），创建yml资源文件，在请求头的accept-language判断语言、通过ms的getMessage获取文本

为什么要用后端实现

DRS有多终端、车端、web、移动端，后端同一返回降低维护成本；

DRS的诊断协议标准化要求后端附加语言描述，前端只负责渲染；

一些DTC故障码、诊断报告、维修建议等业务核心内容需要后端统一管理资源文件；

后端用数据库实现的话可以方便扩展新语言、无需重复部署

先进行了灰度发布和AB测试

高并发优化

改造前：之前的实现逻辑:im服务中当车辆和云端连接后，车辆会上传自己的版本，云端比对最新版本和车端版本，如果不匹配则生成版本同步指令、下发车端版本同步指令信息。车端接收到版本同步信息后去调用file文件拉取最新版本
问题:当服务器配置很低时，如果版本发布比较频繁，频繁的版本更新会对 file服务造成压力，故要基于mq削峰；服务器带宽有限，网络资源被打满【瞬时最高100QPS，需要削峰，服务器只能处理12QPS，需要限流】

改造后：版本不匹配时将指令放到rabbitmq、

消费者从RabbitMQ拉取消息，检查车辆在线状态（通过IM服务查询）。若车辆在线，调用IM服务下发升级指令；若离线，延迟重试或记录日志。

延迟消息是指消息在投递到队列之前，会等待一定的时间。只有在时间到达后，消息才会被消费者处理。

将订单信息放到没有消费的队列中，并设置TTL10min、10min钟后没有被消费的消息会被转移到死信队列（要绑定死信交换机），专门用于消费TTL到期、消息被拒绝、队列满了的消息

然后从死信队列中拿出订单看是否支付，未支付则取消订单，恢复库存

发文幂等性

2025年02月15日 留言

为什么要幂等性

客户端对同一接口发起多次相同的请求时，服务端只执行一次操作

当网络延迟时、用户重复点击时可能会发送多次相同的博客

如何实现

【唯一标识符】

前端按钮置灰、但只从用户体验角度避免重复提交，并没有从系统设计层面完全解决幂等性

后端TOKEN机制：用户进入博客编辑界面客户端会像服务端发送一个获取token请求（UUID）设置过期时间3分钟防止无效token太多

当用户点击发送博客时，请求头携带token到服务端，服务端从redis校验是否存在，不存在setnx成功并发送成功，存在则发送失败

为什么用AOP

将业务细节和幂等性控制分离、修改博客时也用到幂等性，实现代码复用、点赞、评论也能用到，便于横向扩展

其他方案

1、用数据库唯一约束去判断是否存在【唯一标识符】

2、分布式锁token机制，用户发起

3、状态机控制，状态只能单向进行（比如订单只能从未支付到已支付，适用于订单、审批）

4、MVCC

5、消息队列去重

6、前端控制

版本号或时间戳机制

数据库中存储版本号或时间戳，当客户端第一次请求时回传版本号；

当服务端执行操作时比对版本号，如果数据库中记录与版本号一致则是首次请求并更新版本号【乐观锁解决并发安全问题】；

update vId set Vid =vId+1 where vId =#{}

如果不一致则非首次操作，服务端拒绝执行
